# TDD KPI 운영 가이드

이 문서는 `plan_TDD.md`의 KPI 항목(품질/속도/도입)을 실제로 측정하기 위한 최소 운영 기준이다.

---

## 1. 측정 항목

### 품질

- 전체 테스트 통과율: `make test` 성공/실패 비율
- 회귀 버그 재발률: 동일 원인 버그 재오픈 비율
- 플래키 비율: 동일 커밋 재실행 시 테스트 결과 불일치 비율

### 속도

- `make test` 실행시간 (p50/p95)
- CI 피드백 시간: push/PR 후 필수 job 완료까지의 시간
- Red -> Green MTTR: 실패 테스트 발생부터 재통과까지 걸린 시간

### 도입

- 테스트 동반 PR 비율: 코드 변경 PR 중 테스트 변경 포함 비율
- TDD 체크리스트 준수율: PR 템플릿 체크 완료 비율
- AI 가드레일 위반 건수: 우회/완화 패턴 탐지 건수

---

## 2. 수집 명령 (로컬/CI 공통)

```bash
# 전체 회귀
make test

# 기능별 빠른 확인(선택)
make test-module_oxidation
make test-integration_pipeline

# 정적 분석
make typecheck
make lint

# 커버리지 보고
make test-cov
```

CI에서는 아래 형식의 산출물 저장을 권장한다.

- `pytest` JUnit XML (`--junitxml`)
- `coverage.xml`
- lint/typecheck 로그

---

## 3. 목표값 (권장)

| 기간 | 품질 목표 | 속도 목표 | 도입 목표 |
|---|---|---|---|
| 2주 | 전체 테스트 통과율 >= 95% | `make test` p95 10분 이내 | 테스트 동반 PR >= 60% |
| 1개월 | 회귀 버그율 20% 감소 | CI 피드백 15% 단축 | 테스트 동반 PR >= 75% |
| 3개월 | 회귀 버그율 40% 감소 | Red->Green MTTR 4시간 이내 | 테스트 동반 PR >= 90% |

---

## 4. 운영 루틴

1. PR 단위로 체크리스트/명령 실행 결과를 기록한다.
2. 주 1회(권장: 금요일) KPI를 집계한다.
3. 목표 미달 지표 1~2개만 선정해 다음 주 개선 액션을 등록한다.

---

## 5. 해석 시 주의사항

- 커버리지는 단독 지표가 아니다. 통과율/회귀율/피드백 시간과 함께 본다.
- 플래키 테스트는 품질/속도 지표를 동시에 왜곡하므로 우선 정리한다.
- 단기적으로 테스트 수가 늘면 속도가 약간 느려질 수 있으나, 회귀 감소 효과를 함께 평가한다.
